Custom Errors & Revert

How Solady reverts from Yul with 4-byte selectors and tight control flow.

From Solidity error to Yul revert

In Solidity you write:

error InsufficientBalance();

At runtime a revert carries the selector bytes4(keccak256("InsufficientBalance()")) = 0xf4d678b8. Solady skips the compiler and writes it directly:

if gt(amount, fromBalance) {
    mstore(0x00, 0xf4d678b8)   // `InsufficientBalance()`.
    revert(0x1c, 0x04)
}

Primary source: Solidity custom errors.

Why revert(0x1c, 0x04)?

mstore(0x00, selector) writes a full 32-byte word big-endian. The 4-byte selector sits at mem[0x1c..0x1f] — the last 4 bytes of the word. revert(0x1c, 0x04) returns exactly those 4 bytes to the caller.

Vyper’s @revert with error data follows the same ABI rule: 4-byte selector first. In Yul you build that blob by hand.

Control flow in Yul

Solady uses inverted comparisons — revert on failure, fall through on success:

// Solidity: require(fromBalance >= amount)
if gt(amount, fromBalance) { … revert … }

// Solidity: if (allowance != type(uint256).max) { spend }
if add(allowance_, 1) {
    if gt(amount, allowance_) { … revert … }
    sstore(allowanceSlot, sub(allowance_, amount))
}

The add(allowance_, 1) trick: when allowance_ is type(uint256).max, addition wraps to 0 (falsy) — skip spending logic. Cheaper than an explicit max check.
