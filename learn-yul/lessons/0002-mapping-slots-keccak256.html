Lesson 0002

Mapping Slots & keccak256

Derive balance, allowance, and nonce slots by hand — the core skill for reading Solady ERC20.

← All lessons · ← 0001 · Storage slots

The rule you already know

In Solidity, a mapping does not occupy a contiguous run of slots. The base slot holds nothing useful by itself; each key’s value lives at: 

slot(key) = keccak256(abi.encode(key, baseSlot))

Nested mapping allowances[owner][spender] nests the formula: 

innerBase = keccak256(abi.encode(owner, baseSlot))
valueSlot = keccak256(abi.encode(spender, innerBase))

Solady replaces compiler-assigned baseSlot (0, 1, 2…) with fixed seed constants so inherited storage never collides. The math is identical — only the base identifier changes.

Vyper maps work the same way at the EVM level. If you’ve ever wondered where a balance “really” lives on chain, it’s always a keccak256 away from a namespace slot.

Three Solady patterns — side by side

Mapping Seed constant Keys Hash length
balanceOf _BALANCE_SLOT_SEED owner 0x20 (32 bytes)
allowance _ALLOWANCE_SLOT_SEED owner, spender 0x34 (52 bytes)
nonces _NONCES_SLOT_SEED owner 0x20 (32 bytes)

Pattern A — single key (balance & nonce)

From Solady balanceOf:

mstore(0x0c, _BALANCE_SLOT_SEED)
mstore(0x00, owner)
result := sload(keccak256(0x0c, 0x20))

nonces(owner) is the same shape with _NONCES_SLOT_SEED.

Why start the hash at 0x0c?

0x0c = 12 decimal = the ABI padding before a 20-byte address in a 32-byte word. mstore is big-endian: the MSB of the word is at the lowest memory offset.

Execute in order — seed first, owner second:

Step 1 — mstore(0x0c, seed) mem: 0x0c ───────────── 0x27 │ 0x28 ── 0x2b [ 28 zero bytes ] [ seed 4B ] Step 2 — mstore(0x00, owner) (overwrites only 0x00..0x1f) mem: 0x00 ── 0x0b │ 0x0c ─────── 0x1f │ 0x20 ── 0x27 │ 0x28 ── 0x2b [ 12 zero ] [ address 20B ] [ 8 zero ] [ seed 4B ] └ owner mstore ──────────┘ └── untouched from step 1 ──┘ Step 3 — keccak256(0x0c, 0x20) hashes exactly: [ address 20B │ pad 8B │ seed 4B ] = 32 bytes → storage slot

Hashing from 0x00 for 32 bytes would include leading padding but drop the seed. Hashing 64 bytes from 0x00 is the standard abi.encode path — correct but costlier. Starting at 0x0c is the gas-optimized overlap trick.

Pattern B — two keys (allowance)

From Solady allowance:

mstore(0x20, spender)
mstore(0x0c, _ALLOWANCE_SLOT_SEED)
mstore(0x00, owner)
result := sload(keccak256(0x0c, 0x34))
After all three mstores (spender, seed, owner): mem 0x00..0x1f: owner word [12 pad | owner 20B] mem 0x20..0x3f: spender word [12 pad | spender 20B] mem 0x28..0x2b: seed tail (from seed mstore at 0x0c, untouched by owner) keccak256(0x0c, 0x34) — 52 bytes from 0x0c through 0x3f: owner address + padding + full spender word + seed tail = nested mapping(owner → spender) in one hash

Fifty-two bytes = 32 bytes from the owner region (starting at byte 12) plus the full 32-byte spender word. This is the nested mapping formula folded into one hash — no abi.encode call at runtime.

Pattern C — packed single key (transfer hot path)

When from is already on stack, Solady saves one mstore:

let from_ := shl(96, from)
mstore(0x0c, or(from_, _BALANCE_SLOT_SEED))
fromBalanceSlot := keccak256(0x0c, 0x20)

or(shl(96, from), seed) at mstore(0x0c, …) builds the same 32-byte preimage in one write: seed at mem[0x28..0x2b], address at mem[0x0c..0x1f] (via shifted word layout).

Clean vs pack — don’t mix them up

IdiomPurposeWhen
shr(96, shl(96, addr)) Canonical address in low 160 bits permit, eq with ecrecover
shl(96, addr) Address in high 160 bits for or with seed _transfer, transferFrom slot lookup

Pattern C does not need shr(96, shl(96, from)) first: a typed Solidity address is already clean, and shl(96, …) only pulls from the low 160 bits anyway. Adding clean-before-pack is redundant here.

Direct slot (totalSupply)

Not every value is a mapping. Total supply uses a fixed slot — no hash:

result := sload(_TOTAL_SUPPLY_SLOT)   // 0x05345cdf77eb68f44c

When you write your own ERC20 in Yul, decide upfront: which values are direct slots vs hashed mappings.

Trace it yourself

Given owner = 0x1111…1111 (20 bytes), mentally execute Pattern A:

  1. After both mstores, what 32 bytes does keccak256(0x0c, 0x20) see?
  2. That digest is the storage slot number — sload reads the balance there.
  3. Change only the seed constant — same owner maps to a different slot (different mapping namespace).
Checkpoint Open ERC20.sol and find all three hash lengths: 0x20, 0x34, and the direct sload. You should recognize each before moving on.
