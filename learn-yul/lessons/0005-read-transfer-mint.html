The four-phase pattern

Every Solady token movement follows the same skeleton:

1. COMPUTE SLOT → mstore scratch + keccak256 2. LOAD + CHECK → sload, if gt … revert 3. STORE → sstore updated balances (and supply for mint/burn) 4. EMIT → mstore amount + log3

No new opcodes — this lesson wires lessons 0001–0004 together.

transfer(to, amount) — public entry

Uses two-mstore slot pattern (Pattern A from lesson 0002):

Phase 1 — sender slot: mstore(0x0c, SEED); mstore(0x00, caller()); keccak256(0x0c, 0x20)
Phase 2 — check: sload balance; if gt(amount, fromBalance) revert InsufficientBalance
Phase 3a — debit: sstore(fromSlot, sub(fromBalance, amount))
Phase 1b — recipient slot: mstore(0x00, to); keccak256(0x0c, 0x20) (seed still at 0x28..0x2b)
Phase 3b — credit: sstore(toSlot, add(sload(toSlot), amount))
Phase 4 — emit: mstore(0x20, amount); log3(…, caller(), shr(96, mload(0x0c)))

_transfer(from, to, amount) — internal, packed

Same four phases, but Phase 1 uses one mstore:

let from_ := shl(96, from)
mstore(0x0c, or(from_, _BALANCE_SLOT_SEED))
fromBalanceSlot := keccak256(0x0c, 0x20)

Same storage slot as two-mstore — fewer writes, same math. Recipient side still uses two-mstore with mstore(0x00, to).

_mint(to, amount)

Supply check: if lt(add(supply, amount), supply) revert TotalSupplyOverflow
Supply store: sstore(_TOTAL_SUPPLY_SLOT, totalSupplyAfter) — direct slot, no hash
Balance credit: same Pattern A slot + sstore(add(sload(slot), amount))
Emit: log3(…, 0, shr(96, mload(0x0c))) — from = address(0)

_burn(from, amount)

Mirror of mint:

  • Debit from balance with InsufficientBalance check
  • Decrement _TOTAL_SUPPLY_SLOT
  • Emit with to = 0, cleaned from as topic
