Yul for Solidity & Vyper Devs

What inline assembly is, why Solady uses it, and the memory patterns you’ll see everywhere.

← All lessons · Opcode cheatsheet · Storage slots

Why Yul exists

Solidity compiles to EVM bytecode. Along the way it may lower through Yul — a typed-ish, low-level IR where EVM opcodes are plain function calls: sload, mstore, keccak256.

You embed Yul inside Solidity with an assembly { … } block. Solady’s ERC20 inlines almost all hot-path logic there because high-level Solidity adds checks, copies, and ABI encoding you don’t always want to pay for.

Vyper has @revert and tight storage access too, but no inline assembly. Think of Yul as writing the Vyper you wish the compiler emitted — by hand.

Primary source: Solidity Yul documentation.

Anatomy of an inline block

Every Solady hot function follows the same skeleton:

function totalSupply() public view returns (uint256 result) {
    /// @solidity memory-safe-assembly
    assembly {
        result := sload(_TOTAL_SUPPLY_SLOT)
    }
}

Syntax you’ll read constantly

  • let x := … — local variable (not persisted on chain)
  • if cond { … } — no else if; nest or use switch for multi-way
  • for { init } condition { post } { body } — C-style loops (rare in Solady ERC20)
  • function f(a, b) -> r { … } — inner Yul functions inside the block
  • Comments: // only

Types collapse to u256 in the EVM dialect. Addresses, bools, and enums are just 256-bit words with conventions on which bits matter.

Scratch memory: the 0x00–0x3f band

Solady’s ERC20 never allocates dynamic memory for mappings. It reuses a fixed scratch zone starting at byte offset 0x00 to build the input to keccak256. 

// balanceOf(owner) — Solady ERC20.sol
mstore(0x0c, _BALANCE_SLOT_SEED)   // step 1: seed at mem[0x28..0x2b]
mstore(0x00, owner)               // step 2: address at mem[0x0c..0x1f]
result := sload(keccak256(0x0c, 0x20))  // hash 32 bytes from 0x0c

mstore(offset, value) writes a full 32-byte word. keccak256(ptr, len) hashes len bytes starting at ptr.

mstore is big-endian. The most significant byte of the word lands at mem[offset]; the least significant at mem[offset + 31]. An address in a word therefore sits at mem[offset + 12] through mem[offset + 31] — exactly 12 bytes of padding, then 20 bytes of address. That is why hashing from 0x0c (12 decimal) works.
Solidity’s “zero slot” and “free memory pointer at 0x40” still exist. Solady’s scratch patterns deliberately overlap 0x00–0x3f — safe here because these view/write functions don’t call out to arbitrary external code mid-scratch.

Storage: sload / sstore

In Solidity you think mapping(address => uint256) balances. In Yul you think slot number: 

slot := keccak256(packedKeyData)
value := sload(slot)
sstore(slot, newValue)

Solady chooses namespaced seeds (_BALANCE_SLOT_SEED = 0x87a211a2) instead of compiler-assigned slot 0, 1, 2… so inherited contracts won’t collide. That’s a design choice you’ll replicate when writing your own ERC20 core.

Addresses as words

An address is 20 bytes but occupies a 32-byte word. The top 12 bytes should be zero. Solady enforces that with: 

owner := shr(96, shl(96, owner))   // clean upper 96 bits
from_ := shl(96, from)             // align address to high bits for or()

shl(96, addr) moves the address to bits 96–255. Solady then ors it with a 4-byte seed at the low end — packing key + namespace into one word before hashing.
