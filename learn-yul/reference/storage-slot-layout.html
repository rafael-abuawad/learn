Storage Slot Layout

Compiler layout vs Solady namespaced slots — big-endian memory edition.

← Lessons · ← Opcodes · Glossary

Remember: mstore is big-endian

Each mstore(p, v) writes 32 bytes: MSB of v at mem[p], LSB at mem[p+31]. A 20-byte address in a word occupies mem[p+12] through mem[p+31] (12 bytes zero padding first).

Solidity compiler layout (baseline)

For a contract declaring:

uint256 totalSupply;                    // slot 0
mapping(address => uint256) balances;   // slot 1 (base)
mapping(address => mapping(address => uint256)) allowances; // slot 2

Slot for balances[owner]:

slot = keccak256(abi.encode(owner, 1))

In Yul scratch form (64-byte hash):

mstore(0x00, owner)    // [12 pad | owner 20B] at mem 0x00..0x1f
mstore(0x20, 1)         // [12 pad | slot 1]    at mem 0x20..0x3f
slot := keccak256(0x00, 0x40)

Solady: 32-byte hash vs 64-byte encode

Approach Hash Gas
Standard encode keccak256(0x00, 0x40) 64 bytes, two full words
Solady overlap keccak256(0x0c, 0x20) 32 bytes, overlap trick

Solady namespaced constants

uint256 private constant _TOTAL_SUPPLY_SLOT = 0x05345cdf77eb68f44c;
uint256 private constant _BALANCE_SLOT_SEED  = 0x87a211a2;
uint256 private constant _ALLOWANCE_SLOT_SEED = 0x7f5e9f20;
uint256 private constant _NONCES_SLOT_SEED = 0x38377508;

Balance mapping (Pattern A)

mstore(0x0c, _BALANCE_SLOT_SEED)   // seed → mem[0x28..0x2b]
mstore(0x00, owner)                 // addr → mem[0x0c..0x1f]
balanceSlot := keccak256(0x0c, 0x20)
Final mem layout (hash region 0x0c..0x2b): [ address 20B @ 0x0c..0x1f | pad 8B @ 0x20..0x27 | seed 4B @ 0x28..0x2b ]

Allowance mapping (Pattern B)

mstore(0x20, spender)
mstore(0x0c, _ALLOWANCE_SLOT_SEED)
mstore(0x00, owner)
allowanceSlot := keccak256(0x0c, 0x34)   // 52 bytes

Packed single mstore (Pattern C)

let from_ := shl(96, from)
mstore(0x0c, or(from_, _BALANCE_SLOT_SEED))
fromBalanceSlot := keccak256(0x0c, 0x20)

Same 32-byte preimage as Pattern A — one mstore instead of two.

Not the same as shr(96, shl(96, from)) (address cleaning). See Lesson 0002.

Design choice summary

Approach Pros Cons
Compiler slots Simple, auditable with standard tools Inheritance can shift slots; extra indirection
Namespaced seeds (Solady) Collision-resistant across inheritance; single-slot totalSupply Must document seeds; tools won’t auto-decode storage
